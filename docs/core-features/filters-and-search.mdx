# Filters and Search

The value of a proxy debugger is not just capture volume. It is how quickly you can reduce a noisy session to the handful of requests that explain the bug.

## Three levels of narrowing

Rockxy gives you three useful layers of focus:

### Sidebar scope

The sidebar has three modes:

- **Browse** — scope traffic by app, domain, path-focused domain subtree, or live signal.
- **Focus** — apply reusable Focus Sets and configure Noise Control.
- **Library** — return to saved and pinned requests.

This is the fastest way to move from “everything” to “the part of the system I care about.”

### Toolbar filters

Use toolbar filters when you want to refine within the current scope:

- protocol,
- AI, Web3 RPC, gRPC, WebSocket, or GraphQL traffic families when detected,
- content family,
- status-code range,
- or a more advanced filter expression.

### Free-text search

Use search when you already know a clue:

- endpoint path,
- host,
- method,
- status code,
- operation name,
- or a value that appears in the request URL.

## Focus Sets

A Focus Set is a named, reusable traffic scope. Open the sidebar's **Focus** mode and choose **Add** beside **Focus Sets** to create one.

A Focus Set can include:

- one captured application;
- a domain and its subdomains;
- a URL path prefix.

It can also exclude a domain or path prefix. All non-empty include conditions must match, while any matching exclusion hides the request.

For example, a “Checkout API” Focus Set can include the Checkout app and `example.com`, then exclude `/health` and `telemetry.example.com`.

When the current sidebar scope already points to an app, domain, or path, a new Focus Set starts with that scope filled in. The editor also suggests values from captured traffic and shows the current match count before you save.

Saved Focus Sets:

- persist across app launches;
- are available in every workspace;
- can be applied, edited, duplicated, expanded to inspect their rules, or deleted;
- keep their active selection separate in each workspace.

Applying a Focus Set only changes which rows are visible. Capture continues, and the underlying transactions remain available to other workspaces.

## Noise Control

Noise Control hides traffic that should remain captured but stay out of the current workspace's working set.

In the sidebar's **Focus** mode, choose **Configure** beside **Noise Control**, then mute either:

- a domain, including its subdomains; or
- a URL path prefix and its child paths.

You can also right-click a domain in **Browse** and choose **Mute Source**.

Muted sources apply throughout the current workspace, regardless of which Focus Set is active. Use **Unmute** for one source or **Unmute All** to restore them.

<Note>
Focus Set exclusions belong only to that Focus Set. Noise Control is workspace-wide. Neither feature deletes traffic or prevents it from being captured.
</Note>

## Practical debugging patterns

### Find one failing API call

1. scope to the app or domain,
2. filter to error status codes,
3. search for the endpoint name or path segment,
4. open the request and inspect timing plus response body.

### Isolate GraphQL traffic

1. scope to the relevant API host,
2. filter to GraphQL or JSON-oriented traffic,
3. inspect operation names and payloads in the request body.

### Isolate AI or Web3 traffic

1. scope to the client app or provider domain,
2. filter to AI or Web3 RPC traffic,
3. inspect the protocol-specific tab for provider, model, stream, tool-call, JSON-RPC method, batch, chain, or error hints,
4. switch back to Headers, Body, Raw, or Timeline for the underlying HTTP evidence.

### Clean up mobile device noise

1. scope to the app or host you care about,
2. mute analytics domains or paths with Noise Control,
3. save a Focus Set when you want to reuse the same investigation scope,
4. save the useful requests so they survive session cleanup.

## Workspace tabs are part of filtering

Workspaces are not just UI tabs. They let you preserve multiple filtered investigations at once.

Examples:

- one tab scoped to app traffic,
- one tab for error-only requests,
- one tab for a single domain under active replay,
- one tab kept clean for HAR export.

See [Workspaces](/features/workspaces).

## When to use rules instead of filters

Use filters when you are trying to *see* the right request.

Use rules when you are trying to *change* what happens to the request.

That distinction keeps the debugging loop clear.
